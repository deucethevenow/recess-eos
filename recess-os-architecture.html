Core Architecture

The Single Front Door Pipeline

Everything enters through the Asana intake form. Seven request types, two pipelines (standard + QA), automated routing via Asana Rules and AI discovery skills.

Pipeline Flow — Intake to Execution

Everything enters through the Asana form. Discovery runs in the Scoping Pipeline. After discovery, items route to one of three destinations: dept boards (small/dept-level), company product backlog (major work), or To-Do (quick solve). Bugs route to existing projects.

After Discovery: Where Things Go

Items leave the Scoping Pipeline immediately after discovery. The pipeline is for active flow only — nothing sits there. Major items become Asana Projects on the Company Roadmap.

Company Roadmap

When: Major, cross-functional, needs PRD
PRD Accepted → Asana Project created → added to Company Roadmap
Sections: Icebox → Prioritized → In Flight → Complete

Dept Board

When: Small, dept-specific, one team can solve
Examples: Report tweak, form field, process change
Dept lead owns it

To-Do

When: Quick solve (< 2 weeks)
Examples: Fix a permission, update a doc
Reviewed in L10

Icebox (skip discovery)

When: Reviewed, not pursuing now
Goes to: Company Roadmap → Icebox section
Revisited quarterly

Each Path in Detail

Feature

Skill: /discovery (full multi-session)
Who runs it: Submitter (usually Deuce)
Time: Multiple sessions
Output: Notion doc in Discovery database
Then: PRD → Approve → Asana Project created → bugs route here

Single Bug

Skill: /bug-triage
Who runs it: Arbind or Leo (engineers)
Time: 5 minutes
Output: Notion bug doc + sized Asana task
Then: Manually routed to the related Asana Project (if one exists)

QA Session (Multiple Bugs)

Skills: /qa-feedback → /receive-qa-feedback → /execute-plan
Who runs it: Submitter collects, engineer processes
Time: Collection 15-30 min, processing 30-60 min
Output: Notion doc → implementation plan → tasks added to related Asana Project
Then: Skips entire pipeline → direct to Active Work

Issue / Opportunity

Skill: /quick-discovery
Who runs it: Submitter
Time: 5-10 minutes
Output: Structured Asana task, sized + routed
Then: Quick-solve → To-Do for L10, or sized → Scoping Pipeline

Request Type Routing

Each request type triggers a different Asana Rule and routes to a specific skill. Features always get full discovery. Bugs split by volume. Issues and Opportunities use quick discovery.

Feature Run Full Discovery Deuce (COO / Product) /discovery (full multi-session) Notion doc → PRD → Feature Project
Single Bug Platform Run Bug Triage Arbind (Recess Platform) /bug-triage (NEW — dedicated bug skill) Notion bug doc + sized Asana task
Single Bug AI/Auto Run Bug Triage Leo (Recess AI Automations) /bug-triage Notion bug doc + sized Asana task
QA Session Multiple bugs Run QA Feedback Collection Deuce (collects) → Arbind/Leo (processes) /qa-feedback → /receive-qa-feedback → /execute-plan Notion doc → impl. plan → Asana tasks (TDD)
New Project Run Full Discovery Deuce (COO / Product) /discovery (full multi-session) Notion doc → PRD → Feature Project
Issue EOS Run Quick Discovery (Issue) Deuce (COO) — routes to dept lead /quick-discovery (5 Whys + pain rating) Quick-solve → To-Do, or sized → pipeline
Opportunity EOS Run Quick Discovery (Opportunity) Deuce (COO / Strategy) /quick-discovery (value + effort estimate) Sized → pipeline or parked

QA Session Workflow — The Parallel Pipeline

QA sessions bypass the Scoping Pipeline entirely. They have their own collect → plan → approve → execute flow documented in Notion.

sequenceDiagram participant S as Stakeholder
(Deuce/Katie) participant N as Notion participant E as Engineer
(Arbind/Leo) participant A as Asana Note over S,A: Phase 1: Collection (/qa-feedback) S->>S: Walk through feature, find bugs S->>N: /qa-feedback creates Notion doc S->>N: Add items 1 by 1 with screenshots N->>N: Each item: Bug or Enhancement + severity Note over S,A: Phase 2: Processing (/receive-qa-feedback) E->>N: Read QA doc, get all items E->>E: Research codebase for each item E->>E: Flag already-fixed items E->>N: Create implementation plan (v1) N->>N: Before/After for each fix + AC Note over S,A: Phase 3: Review S->>N: Review plan, request changes E->>N: Update plan (v2, v3...) S->>E: Approve plan Note over S,A: Phase 4: Execute (/execute-plan) E->>A: Create Asana tasks (eng + QA subtasks) E->>E: TDD: test first, then implement E->>A: Mark tasks complete S->>S: Verify fixes in QA subtasks
Why QA sessions skip the pipeline: The /receive-qa-feedback implementation plan IS the equivalent of a PRD — it has before/after descriptions, acceptance criteria, and codebase research. No need to duplicate this work in the Scoping Pipeline. Tasks are created during /execute-plan, not during planning.
